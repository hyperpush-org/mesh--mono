# Knowledge

- `reference-backend` startup is stable on the non-empty `DATABASE_URL` path when env validation stays local to `main.mpl` and uses `Env.get` plus `Env.get_int`; the mechanical regression surface is `cargo test -p meshc e2e_reference_backend_runtime_starts --test e2e_reference_backend -- --ignored --nocapture`, and the missing-env check remains `env -u DATABASE_URL PORT=18080 JOB_POLL_MS=500 ./reference-backend/reference-backend 2>&1 | rg "DATABASE_URL"`.
- `meshc migrate` currently parses as `meshc migrate [DIR] [COMMAND]`, so project-scoped calls must be ordered like `cargo run -p meshc -- migrate reference-backend status` and `cargo run -p meshc -- migrate reference-backend up`, not `migrate status reference-backend`.
- In this worktree/runtime, opening a PostgreSQL pool succeeds but the first DB query/execute path in a compiled Mesh binary aborts the process (`Pool.execute`, `Repo.execute_raw`, and `Repo.query_raw` all reproduced it), which blocks both `meshc migrate` execution and the `reference-backend` `/jobs` handlers until the runtime bug is fixed.

  <behavior>
    The infix operator branch in `expr_bp` currently calls `expr_bp(p, r_bp)` immediately after consuming the operator token. If the operator is `|>` or `|N>` AND is at the top level (newlines are significant), a NEWLINE token will be the next token, causing `lhs(p)` to produce a parse error.

    Fix: after `p.advance()` (consuming the `|>` or `|N>` token), when `current` is PIPE or SLOT_PIPE AND `p.is_newline_insignificant()` is false, call `p.skip_newlines_for_continuation()` before `expr_bp(p, r_bp)`.

    Cases that must parse correctly after this change:

    Trailing-pipe (new):
    ```
    x |>
      f()
    ```

    Trailing slot-pipe (new):
    ```
    x |2>
      f(1)
    ```

    Leading-pipe (existing — must not regress):
    ```
    x
      |> f()
    ```

    Mixed trailing + leading in same chain (new):
    ```
    x |>
      f()
      |> g()
    ```

    All forms must produce the same AST as their single-line equivalents:
    - `x |> f()` for trailing/leading `|>`
    - `x |2> f(1)` for trailing/leading `|2>`
  </behavior>
  <implementation>
    In `expr_bp` in `compiler/mesh-parser/src/parser/expressions.rs`, locate the infix operators block (around line 157-175). After `p.advance()` (which consumes the operator) and before `expr_bp(p, r_bp)`, add:

    ```rust
    // ── Trailing-pipe newline skip ──
    // If the operator just consumed is |> or |N> and we are outside
    // delimiters (newlines are significant), skip any newlines before
    // parsing the RHS. This allows: `x |>\n  f()`.
    if matches!(current, SyntaxKind::PIPE | SyntaxKind::SLOT_PIPE)
        && !p.is_newline_insignificant()
    {
        p.skip_newlines_for_continuation();
    }
    ```

    `is_newline_insignificant` is currently `fn` (not `pub(crate)`). Either:
    - Change visibility to `pub(crate)` in `mod.rs`, OR
    - Add a new `pub(crate) fn at_top_level(&self) -> bool { !self.is_newline_insignificant() }` method, OR
    - Inline the check: `p.paren_depth == 0 && p.bracket_depth == 0 && p.brace_depth == 0` if those fields are pub(crate).

    Use whichever approach requires fewest changes. Check field visibilities in `mod.rs` before deciding.
